# Run kind & worktree lifecycle

## Goal

The `run` session kind works end-to-end with the Auto permission preset, demoable headless through the orchestrator's RPC/WS surface with no new UI. A run RPC on a Ready story creates the story's worktree (`~/.helm/worktrees/<repo>/<story-id>/`, sparse checkout excluding `.helm/board/`, `.knowledge/architecture/board-storage.md` §Worktrees) and its `branch` from frontmatter, spawns the run with the brief as its prompt and the brief-body hash snapshotted onto a new `runs` frontmatter entry, streams the session's events over the existing session WS channel, commits its work on the branch, reports its outcome through the Stop-hook backstop, and flips the card to Review or Blocked (`.knowledge/product/features/runs.md` §Run lifecycle).
## Approach

**Entry point.** A new `run.start` RPC (`src/worker/routes/run.ts`) drives the lifecycle from a new `src/server/services/runs.ts` service: fresh read inside the write queue, `canTransition(ready → running)` validation, then worktree → spawn → frontmatter. The validation also requires gate freshness, which the transitions table only checks on `to: ready`: a hand edit to a Ready card's body is legal and stales the verdict while the status stays `ready`, and an uncertified brief must not run — so `run.start` (the initial validation and the init re-check alike) additionally requires `verdictValid`, rejecting with a "brief edited since the gate; re-run the ready gate" error. The note-excluding hash (below) keeps run notes from ever false-positiving this check. The runs service holds its per-story run state — starting guard, init-write promise, live process handle — **for the run's whole lifetime, not just start-to-init**: `run.start` rejects while the story has a live run process, and the state clears only after the close write completes (or the start fails). That closes the otherwise-legal double-run sequence — mid-run move `running → blocked` (the process stays alive), `blocked → ready` (the unchanged body keeps the verdict valid), `run.start` again — which would safety-commit run 1's in-flight work as leftovers and spawn a second Auto session into the same worktree; a story whose card was moved out of `running` mid-run re-runs only after its process closes. Between validation and the `system/init` write the story still reads `ready` on disk (worktree checkout plus spawn-to-init sit in between), which is why the guard exists from acceptance: a second concurrent `run.start` is rejected while the write queue is never held across checkout or spawn. The spawn-to-init wait is bounded: if `system/init` hasn't arrived within a timeout (60s), the child is killed, the start fails with that error, and the guard clears — a wedged CLI (auth stall, network hang) costs one failed `run.start`, never the story's run path, which matters here because a pre-init process has no session id and is unreachable through `session.kill`. `story.move` starts rejecting `to: running` the way it already delegates `ready` to the gate: a bare status write without a spawned process is the same hazard as the hand-typed `running` the storage rules ban. Runs spawn directly in this story; dispatcher/queue routing is 002-04.

**Worktree lifecycle** (new `src/server/worktrees.ts`, shelling out to git like the rest of the repo). The branch is the durable artifact and the worktree is disposable (board-storage §Worktrees), so `run.start` converges on branch-plus-worktree from whatever mix exists instead of assuming two clean states. The branch name comes from `branch` frontmatter, minted `helm/<id>-<slug>` and written back only when the field is absent — a frontmatter name whose ref is gone (this story's own card today) just means the ref is created rather than re-minted. The sequence: `git worktree prune` first, clearing stale registrations left by an out-of-band directory delete; then, if the worktree directory is missing, `git worktree add` at `~/.helm/worktrees/<repo>/<story-id>/` — with `-b <branch> <mainBranch>` when the branch ref doesn't exist, plain `<branch>` checkout when it does (wiped worktree dir, machine move) — and a fresh worktree gets the non-cone sparse checkout excluding `.helm/board/` while keeping the rest of `.helm/`, so the run still loads Helm's rules but board state never enters the branch. An existing worktree is reused after a `git status` check that safety-commits any leftovers before spawning — so run n+1 always starts from a clean tree and never mixes run n's partial work into its own commits — **and a branch check**: reuse verifies the worktree's HEAD (`git rev-parse --abbrev-ref HEAD`) equals the frontmatter `branch`, because a hand-renamed `branch:` or a manual `git switch` in the worktree is legal (only the run is denied branch switching) and would otherwise strand run n+1's commits on a stale branch the frontmatter-driven 002-06 merge never reads. On a mismatch the sequence converges rather than rejects: the leftovers commit lands first on the old HEAD, where that work actually happened, then the frontmatter branch is checked out (created from `mainBranch` if its ref is missing) before spawn. No deletion here (002-07).

**Spawnable `run` row** (`src/sessions/kinds.ts`). The row gets its tools as the Auto allowlist constant — file-edit tools, Read/Grep/Glob, and git Bash patterns covering both the read-only queries and the branch-local writes the run contract demands (`git status/diff/log`, `git add`, `git commit`, `git mv`, `git rm`; no push, no branch switching); the `.helm/` override file is 002-02's. **Two permission-rule mechanics are spike-verified before building, the same treatment as the Stop-hook fact below, because nothing in the repo exercises them and both are load-bearing:** the Bash prefix-wildcard syntax (documented as `Bash(<cmd>:*)`; a space-star form would be an exact-match literal that never fires, silently denying every argument-carrying `git commit -m …` and check-command variant, with the safety commit masking the miss as `chore(helm)` leftovers instead of Conventional Commits), and settings-file path-rule anchoring for the deny rules (relative patterns resolve against the settings file's own directory, and the per-spawn file does not live in the worktree, so the `.helm/` deny is expected to need the absolute form). The spike also settles whether repeated `--allowedTools` flags avoid the CLI's comma-splitting of a single argument — the future loosening path for the comma constraint below. The verified forms are what land in the constant and the settings file; the acceptance run's zero-prompt Conventional Commit doubles as the regression check that the git allowlist actually fires. The managed repo's check command comes from a new **optional** `checkCommand` field on the repo entry in `helm.config.json` (`src/server/config.ts` schema, mirrored in `helm.config.example.json`) — optional so every existing gitignored config keeps booting. The field is constrained to **one simple command**: the schema rejects shell operators (`&&`, `||`, `;`, `|`) **and commas**, both noted in `helm.config.example.json`. Operators break because the CLI checks each component of a compound command against the allowlist separately; commas break in the delivery channel — the runner joins the allowlist into one comma-separated `--allowedTools` argument the CLI splits on commas, so a routine `pnpm --filter a,b test` would shred its own pattern into garbage and deny every self-test, precisely the silent failure the constraint exists to prevent. The registry constant cannot see runtime config, so `spawnSessionProcess` gains a per-spawn `extraTools` option merged into `--allowedTools`; the runs service maps the command to two patterns there — the exact form `Bash(<cmd>)` plus the spike-verified prefix-wildcard form — so the model's natural variants (appended arguments, `2>&1`) run free while unrelated commands stay denied. The command's text must also reach the run itself — the config is the orchestrator's, invisible from the worktree, and static registry text cannot name it — so `runPrompt` states it explicitly ("the repo's check command is: `<cmd>`") alongside the allowlist pattern, and states that none is configured when the field is absent. Absence therefore degrades to a run that knows it cannot self-test, never one guessing `npm test` into loud denials. runs.md §Permission presets words the canonical list as "read-only git", contradicting its own §Run lifecycle ("a finished run commits its work on the story branch"); that doc line is corrected to include branch-local git writes. Board tools are `["update_card"]` only (`ask_user`/needs-input is 002-02). The system prompt is the run contract: implement the brief, commit as Conventional Commits on the story branch, note decisions and progress through `update_card`, never edit `.helm/` files. The prompt is the brief (card body) via a new `runPrompt` in `src/sessions/prompts.ts`; cwd is the worktree; `--permission-mode default` stays, so a non-allowlisted tool call is denied loudly instead of prompting — Auto's headless semantics.

**Spawn plumbing.** The sessions service grows a run-spawn path sharing the existing plumbing (MCP token binding, live/interrupted maps, session-channel broadcast) but with a per-spawn cwd (the worktree), hook settings, and the `extraTools` merge; `src/sessions/runner.ts` grows the `--settings` (Stop-hook config) and `extraTools` options, and run spawns are **detached as their own process-group leaders**, so a group kill can reach not just the `claude` process but the tool subprocesses it spawned (a still-running build or test), which a single-pid signal would miss. **Every kill of a run uses the same bounded group escalation** — `session.kill`, the retraction abort, and the spawn-to-init timeout alike signal the process group (negative-pid SIGTERM, 5s wait, SIGKILL, short wait), never the single pid, because the brief's own premise is that pid-death doesn't quiet the tree; a chat kind's `child.kill()` keeps the plain single-pid SIGTERM it has today. `messageSession` starts rejecting `run` sessions the way it rejects the always-cold kinds: a resume from the chat entrypoint would run in the main checkout — session lookup is cwd-scoped, so the resume dies stale, and the stale-reseed branch would then fresh-spawn a write-capable, Auto-allowlisted run session in the main working tree, outside the run lifecycle (no run entry, no worktree, no Stop hook, no safety commit). Steering resumes land with 002-03 through a run-aware path. Runs stay in the `live` map, so `session.kill` keeps working on them.

**Frontmatter writes** (single writer, all through `enqueueWrite`). On `system/init`: one write sets `status: running` and appends the run entry `{n, session, brief: briefHash(body), started}` — the snapshot review later grades against. The init write re-reads the story inside the queue and re-validates `ready → running` rather than writing unconditionally: `ready → refining`/`ready → blocked` moves and hand edits stay legal during the seconds-long start window, and the user's retraction wins — on a failed re-check the start aborts, the spawned process is killed, the guard clears, and no run entry is written. The close path arms only once that `running` write lands, and arming is serialized against the queued write rather than assumed: the runs service keeps the per-story init-write promise beside the starting guard, the close handler awaits it before deciding armed vs bypass, and the init write's in-queue re-check also aborts (writing nothing) when the process has already exited — so an early death or `session.kill` racing the queue can never land `status: running` plus a run entry for a dead process. An exit before the write lands — the retraction abort's kill, or a process that dies between spawn and `system/init` — bypasses the close handling entirely (guard cleared, no run entry, no status write; the failure surfaces as `run.start`'s error or a logged start failure), so an aborted start can never funnel into the Blocked path below. After arming, on close: **first confirm the process group is dead** — on any non-natural exit the close handler runs the same bounded escalation before touching the tree, so the safety commit never races a still-writing build subprocess; a group that survives SIGKILL logs, skips the commit, and parks Blocked with the error, mirroring reconciliation — then safety commit, then a queue write that re-reads the story (the stated analogue of the init re-validation, since moves out of `running` stay legal mid-run). When the story still reads `running`, a success `result` writes `status: review` with `outcome: review` plus tokens/minutes from the result event, and anything else (error result, crash, `session.kill`) writes `status: blocked` with `outcome: blocked` and the last error on a new optional `error` field on `runSchema` (`src/board/schema.ts`; board-storage.md's run-entry example gains the field). When a mid-run move or hand edit already parked the card elsewhere, the user's move wins: no status is written, but the run entry still records `outcome`/`tokens`/`minutes`/`error` — updating an entry is bookkeeping, not a transition — and the safety commit has already preserved the work on the branch.

**Safety commit.** After any exit, a dirty worktree is committed on the story branch with an orchestrator-identifiable message (`chore(helm): commit run leftovers`), so review sees everything and a Blocked card's partial work survives the disposable worktree. Orchestrator commits are bookkeeping, not the run's Conventional Commits: every safety-commit site passes `--no-verify --no-gpg-sign`, so the managed repo's pre-commit hooks never fail on half-finished leftovers and a `commit.gpgsign` pinentry never hangs a headless flip. If the commit still fails, one rule covers every site — the flow never proceeds as if the tree were clean: the close path parks the card Blocked with the git error on the run entry instead of flipping to Review; the `run.start` reuse check aborts the start with the error; reconciliation logs it, still parks Blocked, and leaves the dirty tree for the next reuse check to retry. The `.helm/` bar is structural, not prompt-only, closed at **three** layers: the per-spawn `--settings` file that already carries the Stop hook adds deny rules for file edits under the worktree's `.helm/` (in the spike-verified anchoring form, expected absolute since relative patterns resolve against the settings file's directory), so a confused run's write attempt fails loudly instead of landing; the safety commit's pathspec excludes `.helm/` (`git add -A -- . ':(exclude).helm'`), so a violation that reaches the tree never rides a safety commit; and because the allowlisted `git rm`/`git mv`/`git add` prefix wildcards cannot express path constraints — a run could land `.helm/` changes inside its own commits, touching neither layer — the close path and the `run.start` reuse check verify `git diff <mainBranch>...<branch> -- .helm/` is empty, parking the card Blocked with the violation as the run entry's error when it is not (the same never-proceed-as-clean rule as a failed safety commit). No `.helm/` change reaches 002-06's merge through any of the three paths.

**`update_card`** (`src/server/mcp/tools.ts` + `schemas.ts`, name added to `BOARD_TOOLS`). Direct write through the write queue, no proposal widget — the run's one mutation path. The payload is a short markdown note, and the handler sanitizes it before appending: heading markers and frontmatter fences are stripped or escaped, because `parseBrief`'s last-occurrence-wins section parsing would otherwise let a note carrying a `## Acceptance criteria` (or any template-section) line silently replace the real section for the gate check, review grading, and the UI — inside the note-excluding hash's blind spot, where `verdictValid` cannot catch it. The sanitized note is appended as a bullet under a trailing `## Run notes` section (created on first use). Append-only and heading-free by construction: it cannot touch frontmatter, status, criteria checkboxes, or the body's section structure.

**Note-excluding brief hash.** `briefHash` (`src/board/hash.ts`) hashes the whole body today, so the first `## Run notes` bullet would stale the gate verdict — breaking the blocked → ready → re-run path (`canTransition` requires `gate.brief === briefHash(body)`) — and shift a later run's spawn snapshot even with an untouched brief. The scoping is subtractive, not reconstructive: the hash stays the same FNV over the raw body string, with only the `## Run notes` **block** stripped before hashing — from the final `## Run notes` heading up to the next `##` heading or end of body. Sanitized notes (heading-free by construction) always fall inside that block, while anything a hand edit appends after the section — a new heading, replacement criteria at EOF — starts a new block that re-enters the hash and stales the verdict, preserving the invariant that hand edits stale the gate and that freeform hand-added sections stay part of the hashed contract. A body without notes hashes identically to today, so every recorded `gate.brief`, this story's own included, stays valid with zero migration. The stripping is a string-level helper beside the hash in `src/board/` (the module runs in the browser bundle, so no server imports). board-storage.md records the exclusion alongside the run-entry change.

**Stop-hook backstop, POST leg.** The spawn's settings register a Stop hook that POSTs to a token-addressed orchestrator endpoint (`/hooks/run/<token>`, mounted beside the MCP routes). The hook's payload carries no success/error signal, so the POST is strictly a liveness fact — the turn ended without interruption — never a success verdict on its own; the Review/Blocked branch always keys on the stream's `result` subtype when one was seen. The POST decides anything only when the stream missed the result entirely, and its mapping there hangs on a CLI fact claude-integration.md does not verify (only "fires on normal completion, not on SIGTERM"): whether the Stop hook also fires on an `error_during_execution` turn. Spike-verify that before writing the close handler. If it fires only on clean completions, a POST-only close maps to Review without tokens/minutes; if it fires on error turns too, the POST stays liveness-only and a POST-only close parks Blocked with a "run finished but its result was not observed" error — the work is on the branch either way, and review recovers it. The direct-frontmatter-write fallback is deferred (Out of scope).

**Startup reconciliation** (runs service start). The sweep keys on **pid files first, card status second**, because a hard crash can land anywhere in the lifecycle: reconciliation scans `~/.helm/worktrees/<repo>/` for pid files — each run spawn writes the child pid there (`<story-id>.pid`) before anything else and removes it on close — so an orphan from a crash inside the start window (worktree ready, group live, story still reading `ready` on disk with no run entry) is found and killed even though no card reads `running`; killing it is enough there, since the next `run.start`'s reuse check safety-commits whatever it wrote, and the lifetime guard's job is done by the kill. When a live pid's command line is a `claude` process (guarding against pid reuse), the sweep kills the orphan's **process group** (negative-pid signal; runs spawn as group leaders exactly for this) with a **bounded escalation** — SIGTERM, a short wait (5s), SIGKILL, another short wait — because pid-death alone doesn't quiet the tree: the orphan's own tool subprocesses can outlive it and keep writing, and an unbounded wait would let one wedged process stall the whole sweep, the same one-failure standard the spawn-to-init timeout enforces. A group that survives even SIGKILL logs the failure, skips the safety commit (the tree may still be moving), and still parks the card Blocked when one is `running`, leaving the leftovers for the next reuse check — one wedged orphan costs one story's safety commit, never the sweep. Then, for each story reading `running` at boot: reconciliation parks the card in Blocked and, when an open run entry exists, marks it `outcome: blocked` with error "orchestrator restarted mid-run"; it also owns the crash-path safety commit — the close handler that normally commits leftovers no longer exists (and the Stop hook does not fire on SIGTERM), so before the frontmatter write it commits any dirty state in the story's worktree with the same identifiable message. Reconciliation is defensive per story, because a `running` story at boot need not have the state the happy path leaves behind (hand edits stay legal): a hand-typed `running` with `runs: []` parks in Blocked without inventing a run entry, a missing worktree directory or pid file skips that step, and each story reconciles inside its own try/catch — with every wait bounded, so neither a throw nor a hang escapes — with the failure logged; one bad story never aborts orchestrator boot or the rest of the sweep. Together with the worktree-reuse check at `run.start`, a dead run's partial work always lands on the branch instead of leaking into the next run. Re-entry is the normal blocked → ready → re-run path.

**WS.** Run events ride the existing `session` channel broadcast, already tagged with kind and session id; no channel changes. Session lookup is cwd-scoped and resume-from-recreated-path is spike-verified, so the run's session stays resumable from the worktree for 002-02/03.

## Acceptance criteria

- [ ] `run.start` on a Ready story creates `~/.helm/worktrees/<repo>/<story-id>/` checked out on the story's `branch`; `.helm/board/` is absent from the worktree while the rest of `.helm/` is present (verify with `ls` in the worktree).
- [ ] After spawn, the story file shows `status: running` and a new `runs` entry whose `brief` equals the current brief-body hash and whose `session` matches the id streaming on the WS `session` channel with kind `run`.
- [ ] A tiny real Auto run (one-file trivial brief) completes with zero permission prompts: its commit lands on the story branch, the card flips to Review, and the run entry records `outcome: review` with tokens and minutes.
- [ ] During that run, the Stop hook's POST reaches the orchestrator's `/hooks/run/<token>` endpoint (receipt observable in the orchestrator log).
- [ ] An `update_card` call from the run appends its note under `## Run notes` in the story file; frontmatter, status, and criteria checkboxes are byte-identical before/after.
- [ ] A run exiting with uncommitted worktree changes gets an orchestrator-identifiable `chore(helm): …` safety commit on the story branch before the status flip (`git log` shows it), on failed exits as well as successful ones.
- [ ] `session.kill` on a live run parks the card in Blocked with `outcome: blocked` and an `error` recorded on the run entry.
- [ ] Restarting the orchestrator while a card is `running` parks it in Blocked on boot, with the restart reason as the run entry's `error`.
- [ ] Moving that Blocked story back to Ready and calling `run.start` again reuses the same worktree and branch and appends run entry `n: 2`.
- [ ] `run.start` on a non-Ready story returns `ILLEGAL_TRANSITION`, and `story.move` to `running` is rejected.
## Out of scope

- Guarded/Manual presets, the permission-prompt plumbing, needs-input, and `ask_user` for runs → 002-02. Here the run's board tools are `update_card` only.
- The `.helm/` Auto-allowlist override file → 002-02; this story ships the canonical constant only.
- Activity timeline UI and steering resumes → 002-03; also the mid-run brief-edit notice (editing stays legal and the snapshot semantics already hold — the notice is a 002-03 surface).
- Routing runs through the 001-06 dispatcher, the concurrency cap, and the rate-limit meter/auto-pause → 002-04. Runs spawn directly here; a rate-limit failure parks the card in Blocked like any other error.
- Compaction → 002-05; a window-exhausting run fails to Blocked.
- The review surface, the three exits, and worktree deletion → 002-06/07.
- Board buttons and any UI → 002-08.
- The Stop hook's direct-frontmatter-write fallback → a later hardening story. Known debt, accepted in refinement: a run that finishes while the orchestrator is down parks in Blocked on reboot even though its work is on the branch; review recovers it.
- Deterministic close-handler ordering on graceful stop → the same hardening story. Accepted risk (gate override): the sessions service's stop SIGTERMs live children without awaiting or suppressing the runs close handler, so which of the close write and the process exit wins is nondeterministic — but every end state is recovered by boot reconciliation (the card parks in Blocked and leftovers safety-commit on the branch either way), leaving only the run entry's error string variable.
